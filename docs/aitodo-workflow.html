AI Todo architecture tour

How the app thinks, saves, and repeats tasks.

This page is a guided map of the modules in aitodo: the React screen, the client state layer, the shared task and recurrence rules, the Express API, and the SQLite-backed account workspace.

The Big Picture

AI Todo is a self-hosted, multi-account task manager. The core product objects are Accounts, Tasks, Categories, Templates, and Sessions. The client renders and orchestrates interactions, the server enforces account ownership, and shared pure modules keep task-list and recurrence behavior testable.

Runtime shape

  • ClientReact 19 + Vite 8 + TypeScript
  • ServerExpress 5 with cookie sessions
  • DataSQLite through better-sqlite3
  • TestsVitest, supertest, pure module tests

Module Map

Think of the codebase as four layers: UI components, client orchestration, shared business rules, and server persistence.

src/App.tsx

The top-level screen coordinator. It chooses auth vs. main app, owns visible filters, sort preference, and wires handlers into components.

  • Uses useAuth for session state.
  • Uses useTodoStore for task data.
  • Calls deriveTaskList before rendering.

src/components/

Reusable UI pieces: task rows, calendar, category bar, auth form, sort dropdown, recurrence picker, and task fields.

  • Mostly receives data and callbacks.
  • TodoItem handles nested task UI.
  • CalendarPanel highlights due and projected dates.

src/hooks/

React adapters over app state. Hooks keep component code from knowing HTTP details or pending/error bookkeeping.

  • useAuth: login, register, logout, current Account.
  • useTodoStore: Tasks, Categories, Templates.

src/api.ts

The client HTTP boundary. It wraps fetch, includes cookies, normalizes failures, and exposes typed API helpers.

  • Throws AuthError on 401.
  • Maps SQLite-style done values to booleans.

src/task-workspace.ts

A non-React client workflow core. It decides when to reload lists from the server and when a local patch is enough.

  • Reloads after adds, deletes, recurrence changes.
  • Patches simple field changes locally.

src/task-list.ts

Pure list derivation. It builds the tree, filters by category or date, includes projected recurring dates, and sorts.

  • Depends on utils/tree.
  • Uses recurrence projection for calendar date views.

src/recurrence*.ts

Recurring task behavior. recurrence.ts is client-facing API and labels; recurrence-rules.ts is shared pure math.

  • Validates recurrence configs.
  • Computes next and projected dates.

server.ts

The Express composition root. It creates modules, installs middleware, defines routes, and serves the Vite build in production.

  • Auth routes come first.
  • /api routes require a Session.

server/database.ts

Database schema and migration setup. It creates tables and adds missing columns for existing local databases.

  • Enables foreign keys.
  • Seeds the legacy default Account.

server/task-persistence.ts

The SQLite data access module. Its important boundary is forAccount(accountId), which scopes operations to one Account.

  • Creates Tasks and Categories.
  • Guards parent, category, and template ownership.

server/task-mutations.ts

The patch validator and mutation coordinator. It validates incoming task edits and delegates recurrence completion behavior.

  • Checks priority and date formats.
  • Completing a recurring Task may spawn the next Task.

server/account-session.ts

Account product logic. It registers, logs in, shapes the current Account response, and merges preferences.

  • Hashes passwords with bcrypt.
  • Leaves JWT cookie mechanics to auth.ts.

App Startup Workflow

On a normal local run, the Express API and Vite dev server run together. The browser loads React from Vite, while API calls go to Express.

Start the processes

npm run dev starts tsx server.ts and vite. In development, Express enables CORS for Vite at localhost:5173.

Initialize storage

startServer() opens the SQLite database, calls initDb(db), builds the Express app, and listens on port 3001 unless overridden.

Mount React

src/main.tsx mounts <App />. App asks useAuth whether there is an active Session before showing the task workspace.

Common Workflows

These are the paths worth memorizing when changing behavior.

Add a Task

From form submit to refreshed list

1

App.tsx trims the input and calls addTodo(text, activeCat).

2

useTodoStore wraps the call with pending/error state.

3

task-workspace calls api.createTodo, then reloads Tasks.

4

server.ts handles POST /api/todos through persistence.forAccount(req.userId).

Edit a Field

Small server write, local UI update

1

UI calls a change handler such as changeDueDate or changeTitle.

2

api.patchTodo sends PATCH /api/todos/:id.

3

task-mutations validates the patch and writes through the Account workspace.

4

task-workspace applies a local patch instead of reloading the whole list.

Complete a Recurring Task

Completion can spawn the next occurrence

1

Toggle calls patchTodo(id, { done: !done }).

2

task-mutations delegates completion to recurring-task-operations.

3

nextRecurrenceDate computes the new due date from the Template.

4

The server transaction marks the tree done and inserts the next Task if needed.

Sign In

Session cookie unlocks Account data

1

AuthForm calls useAuth.login or register.

2

account-session validates credentials and returns an Account view.

3

auth.ts sets an httpOnly session cookie with a JWT subject.

4

requireAuth reads that cookie on later /api requests and sets req.userId.

Where To Change What

When you know the kind of change, start in the narrowest module that owns that behavior.

Visual layout or controls

Start in src/components/ and src/App.css. If the control needs new data or actions, then connect it through App.tsx and useTodoStore.

Task filtering or sorting

Start in src/task-list.ts. Tests live in src/task-list.test.ts, and this module is pure enough to test without React.

Client write workflow

Start in src/task-workspace.ts. It owns reload-versus-local-patch decisions, while useTodoStore should stay mostly a React adapter.

API route behavior

Start in server.ts to find the route, then move into the smaller module that owns the rule: mutations, recurrence operations, account session, or persistence.

Database shape

Start in server/database.ts for schema setup, then update server/task-persistence.ts so the rest of the app uses named methods instead of SQL.

Recurrence logic

Start in src/recurrence-rules.ts. It is shared by client projections and server spawning, so changes there should be covered by focused recurrence tests.

Rule of thumb: UI modules should ask for things, workspace modules should orchestrate things, server modules should validate and enforce ownership, and persistence should be the only layer that knows SQL details.

Testing Map

The test files mirror the architecture boundaries. Use the most focused test that can prove the behavior.

  • src/task-list.test.ts verifies derived list behavior: tree, filters, sorts, and date views.
  • src/task-workspace.test.ts verifies client orchestration without React or HTTP.
  • src/recurrence-rules.test.ts verifies recurrence validation, projection, and next-date math.
  • server/*.test.ts verifies API routes and lower-level server modules, including Account isolation.
  • src/components/todo-item-interactions.test.ts covers UI interaction helper behavior.

Useful commands: npm run test, npm run lint, npm run build, and npm run dev.
